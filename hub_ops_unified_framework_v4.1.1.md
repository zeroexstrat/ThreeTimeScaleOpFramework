**Critical caveat — `truth-data.js` is the LTC quiz training table, not a sort-chart lookup.** Its purpose is to teach sorters every *valid* routing decision, which includes:

- **WORMA P / PRORI P** — primary preload trailers (canonical SLIC→PD assignments)
- **WORMA N / PRORI N** — night-sort overflow trailers (catch-all bays for blowout / cross-region overflow)
- **CCHIL S** — multi-belt valid answers (encoded as comma-separated index lists in the quiz)

Naïve ZIP-count majority on the quiz table returns the most common routing across all flow types, not the primary. For SLIC 0169 (WORCESTER), the quiz table shows 1,229 PD-09 rows, 936 PD-11 rows, 240 PD-04 rows, and only 22 PD-03 rows — yet the canonical primary trailer is on PD-03 (WORMA P, bay 75 region). The PD-09/PD-11/PD-04 entries are WORMA N overflow that the quiz teaches sorters to recognize. This is why v4.1.1 anchors on operator ground truth for the highest-utilization SLICs and uses the truth table only to flag CURE destinations that fall *outside* primary PD routing.

## 18. The Bridge v4.0 Asked For

v4.0 introduced Theorem 5.1 (CURE×PD Coupling) by *naming* the function $B: \mathcal{S} \to \mathcal{B}$ that maps destination SLICs to PD belts — but did not construct $B$. The construction requires the ZIP→SLIC→belt truth table that LTC-Twi maintains, and that table was deliberately deferred to a later iteration so v4.0 could focus on the abstract structure of the coupling.

v4.1.1 closes that bridge. With $B$ concretely available, Theorem 5.1's per-zone dock-pressure score becomes a computable artifact that the tracker can render *before* the sort starts — unlocking what v4.0 §14 listed as "pre-sort container mode."

This document presents three contributions:

1. **§19 — The SLIC→PD mapping.** Definition 19.1 constructs $B$ from `pipeline_data.pkl`. Coverage analysis on the 04.29 CURE export: 87 of 103 rows mapped, 16 unmapped, with the unmapped set characterized.
2. **§20 — Pre-sort mode.** The zone Φz prior and headcount formula from v4.0 §14 are restated as functions of $B$ and the pre-staged CURE export, so they can run before 18:00.
3. **§21 — Tracker integration.** Concrete spec for the tracker v2.11 "Pre-Sort Zone Pressure" panel and its companion in the dashboard, with two figures.

## 19. Constructing the SLIC→PD Function $B$

### 19.1 Source data and the WORMA / PRORI distinction

LTC-Twi's pipeline produces two artifacts, but **neither is a direct lookup for "where does SLIC s's primary trailer load tonight":**

- **`pipeline/pipeline_data.pkl`** (intermediate, 345,131 rows) — pre-conflict-resolution expansion of the sort chart's ZIP ranges. Contains overflow-routing entries.
- **`app/js/truth-data.js`** (production, 40,949 rows) — post-conflict-resolution LTC quiz training table. Generated by `pipeline/gen_truth_twilight.py`. Records every *valid* routing decision a sorter needs to make, including overflow.

The canonical "where does the primary trailer for SLIC s live tonight" source is the operational **sort chart** `CHEMA_Twilight_Sort_Chart.xlsx`, read by `process_zips.py`, **plus** the floor coordinator's bay-level knowledge of which trailer at which bay is the primary P versus the night-sort overflow N.

The WORMA / PRORI distinction operationally:

- **WORMA P (Worcester MA Preload).** The primary trailer for SLIC 0169 lives on PD-03 (bay 75 region) and takes the canonical 016xx ZIP catchment. WORMA P is loaded throughout the day sort and twilight sort.
- **WORMA N (Worcester MA Night).** A secondary overflow trailer lives on PD-04 bay 106 during the twilight sort. It receives a *much wider* aggregation: all western Mass ZIPs, CT, and the broader WORMA N catchment set. WORMA N is the safety valve when other belts blow out. Crucially, the LTC quiz table records every ZIP that *can validly go onto* WORMA N, which is why naïve ZIP-count majority for SLIC 0169 returns PD-09 / PD-11 / PD-04 (the overflow destinations sorters need to learn) and not PD-03 (the primary, which sorters learn through bay memorization).
- **SPRMA bay 105 (Springfield MA).** PD-04 bay 105 holds a SPRMA trailer during the day sort that takes western Mass SLICs, finishes some point during twilight, and is then replaced by the WORMA N at bay 106.
- **PRORI P / PRORI N.** The same pattern for Providence RI. PRORI P (SLIC 0299) is on PD-12 (Top Blue) across all three shifts and takes the 028–029 ZIP range. PRORI N is an ad-hoc overflow trailer used for blowouts from HASTX / INDTX / I81IN / TOLOH / LEXKY — packages bound for those destinations are sent to PRORI N when their primary trailer is full and the next sequence trailer isn't yet docked.

\begin{definition}[SLIC-to-belt function — primary trailer assignment]
\label{def:slic-to-belt-v411}
For each SLIC $s \in \mathcal{S}$, the primary-trailer belt $B(s)$ is determined by the following priority chain:

\begin{enumerate}
\item \textbf{Operator ground truth} (Table~\ref{tab:rafael-ground-truth}). For the five highest-utilization SLICs in the CURE 04.29 export — 0169 WORCESTER (PD-03), 0209 NORWOOD (PD-03), 0249 BROCKTON (PD-08), 0219 WATERTOWN HUB (PD-10), 0299 PROVIDENCE HUB (PD-12) — the floor coordinator provides direct bay-level assignment. These are authoritative.
\item \textbf{Sort-chart lookup.} For SLICs not in the ground-truth table, $B(s)$ is read from `CHEMA_Twilight_Sort_Chart.xlsx` (operations document; sibling to truth-data.js).
\item \textbf{Best-effort fallback.} Where neither ground truth nor a clean sort-chart lookup is available, $B(s)$ falls back to the most common belt in `truth-data.js` for SLIC $s$, *with the caveat that this may return an overflow assignment*. The fallback is marked in the output JSON as $\text{src}(s) = \texttt{best-effort}$ and excluded from operational alerting until reviewed.
\end{enumerate}
\end{definition}

## 20. Pre-Sort Mode

### 20.1 The two pieces of information needed before 18:00

To produce Φz priors and headcount targets for the upcoming sort *before* induction begins, the coordinator needs:

1. **Volume forecast per zone.** Either a CURE export from the prior business day (always available by ~10:00 the day of the sort) or an MTA prediction from the historical curve (v1.4 §14, MTA v1.7 normalized-curve + Kalman cold-start).
2. **Headcount per zone given a PPH target.** Volume × hours / PPH-target — the inverse of v3.6 §13.6 Φz computation.

The CURE×PD coupling makes (1) tractable without iGate data, which is essential for pre-sort mode since iGate does not capture volumes until 18:00.

\begin{definition}[Pre-sort zone volume forecast]
\label{def:pre-sort-vol}
Let $C^{(\text{pre})}$ be the most recent CURE export available before the sort starts. The pre-sort volume forecast for zone $Z_i$ is
$$
\hat{V}_{Z_i} := V_{\text{plan,bldg}} \cdot \kappa_{Z_i} \cdot \alpha_{\text{CURE}}(Z_i)
\quad\text{with}\quad
\alpha_{\text{CURE}}(Z_i) := \frac{\bar{U}_{Z_i}(C^{(\text{pre})})}{\bar{U}_{\text{building}}(C^{(\text{pre})})}
$$
where $V_{\text{plan,bldg}}$ is the SOR Volume planned for the building, $\kappa_{Z_i}$ is the asymptotic zone share (Theorem 19.1, v2.1), and $\alpha_{\text{CURE}}$ is the CURE-derived zone tilt factor: how much hotter or cooler this zone is running today versus its long-run average.
\end{definition}

The tilt factor $\alpha_{\text{CURE}}$ is the new ingredient. On any given day the zone shares vary modestly around their structural constants. CURE — captured the prior day — picks up which destinations are at unusually high utilization, and Definition 20.1 translates that into a same-day zone volume tilt.

For 04.29 → 04.30 (hypothetical, since 04.29 is a Wednesday and the next sort would be 04.30 Thursday), with the operator-corrected per-zone aggregation:

| Zone | Mapped CURE dests | Empirical share | $\kappa_{Z_i}$ baseline | $\alpha_{\text{CURE}}$ implied |
|------|------------------:|----------------:|------------------------:|-------------------------------:|
| Z1 (PD01-04)   | 23 | 0.28 | 0.31 | 0.90 |
| Z5-8 (PD05-08) | 35 | 0.43 | 0.31 | 1.38 |
| Z9-12 (PD09-12)| 24 | 0.29 | 0.31 | 0.94 |

The 04.29 tilts read cleanly: Z1 and Z9-12 are running slightly below their structural baselines (Z9-12 about 6% cooler than baseline, Z1 about 10% cooler) while Z5-8 is running 38% hotter — dragged up by the 13 PD-07 destinations (Portland, Stratham, Wells, Mebane Hub, etc.) plus several high-util PD-08 destinations (Bell Hub, Brockton). For a building plan of 120,000 packages on 04.30, the Z9-12 forecast is 0.29 × 120,000 ≈ 34,800 packages — close to the 38,547 observed on 05.04 Monday, with the 9.7% gap explained by the Monday→Wednesday volume difference (γ ≈ 0.94) plus the day-of-week tilt difference between Wednesday and Monday.

### 20.2 Headcount target

Given the volume forecast and a PPH target (per-employee target in the steady-state phase, default 200 from v1.3), Theorem 5.1 in v4.0 specified:
$$
N_{Z_i}^* = \frac{\hat{V}_{Z_i}}{ \text{PPH}_{\text{target}} \cdot H_{\text{sort}} }
$$
For Z9-12 at 39,600 / (200 × 4) = 49.5 sorters — a number the coordinator can use to set borrow/loan requests *before* the sort starts.

## 21. Tracker / Dashboard Integration

### 21.1 New tracker panel — "Pre-Sort Zone Pressure"

Position: in the Hub tab, above the Pace panel, visible only when (a) sort is in Phase 0 *or* the SOR Volume actual is still zero. Three rows of cards (one per zone), each showing $\bar{U}_{Z_i}, P_{Z_i}, U_{Z_i}^{\max}$, the top 3 destinations by utilization, and the derived headcount target $N_{Z_i}^*$.

```javascript
// Pseudocode — invocation from dopRenderAll, gated on Phase 0 or zero-actual.
function _renderPreSortZonePressure(state, phaseInfo){
  if (phaseInfo.phase !== 0 && state.netVol > 0) return;          // hide once live
  const cure = DATA.cure;                                         // already parsed
  const slic2belt = await loadSLICToBelt();                       // from truth-data.js
  const zones = computeZonePressure(cure, slic2belt);
  // zones = {Z1: {meanU, sumP, maxU, top: [...]}, Z2: {...}, Z3: {...}}
  renderZoneCards(zones, state.planVol, PPH_TARGET, HOURS);
}
```

The SLIC→PD mapping is a static lookup that the tracker can ship as a pre-computed JSON or embed at build time. ~54 rows in the current CURE-relevant subset, so it adds negligible filesize. Refresh quarterly when the LTC-Twi truth table updates.

### 21.2 New dashboard panel — "Zone Pressure Timeline"

In the post-sort dashboard, show $P_{Z_i}$ for each zone across the 5-sort week (or 4-week trailing window when more data is available). A zone running consistently above 1700 sum-pressure (the level Z5-8 sat at on 04.29) is a candidate for permanent borrow/loan rebalancing.

## 23. Limitations

- The mapping $B$ is *static* per release. ZIP→SLIC routing changes occasionally (network reorganizations, new shipper hubs). The pipeline should be re-run weekly; the truth table version should be stamped on the tracker's About panel.
- $\alpha_{\text{CURE}}$ is a single-snapshot tilt. Variance across CURE snapshots within a day is not captured here. v4.2 will use multiple intra-day CURE pulls when the Tailscale watcher is in place.
- The unmapped 16% of CURE rows is a real coverage gap, especially for late-week sorts when air-gateway volume becomes a larger share of total outbound. The action item to LTC-Twi addresses this directly.
